บทที่ 4

Loop + Selenium หน้างานจริง

✅ SeleniumLibrary 6.x

🎯 บทนี้จะได้อะไร

  • วน loop ผ่าน element บนหน้าเว็บที่จำนวนไม่แน่นอน
  • ตรวจ table ทีละแถวโดยไม่ต้อง hardcode locator
  • กรอก form หลายแถวจาก data
  • รอ spinner / loading ด้วย WHILE แทน Sleep แบบ fixed
  • วน pagination จนหมดทุกหน้า

ถ้าเรียน 3 บทก่อนมาแล้ว บทนี้ไม่มีอะไรใหม่ด้านภาษา — แค่เอา FOR/WHILE ไปใช้กับ Selenium จริงๆ keyword ใหม่ที่เจอมีแค่ 2-3 ตัว จะอธิบายตรงจุดที่ใช้

Pattern 1

วน element ที่ไม่รู้ว่ามีกี่ตัว

ใช้บ่อยสุดเลย — เมื่อหน้าเว็บ render list หรือ card ที่จำนวนขึ้นอยู่กับ data เช่น รายการสินค้า, notification, ผลการค้นหา

keyword ใหม่
Get WebElements locator — คืน list ของ element ทุกตัวที่ match locator นั้น แต่ละตัวเป็น WebElement object ที่เอาไปใช้กับ keyword อื่นต่อได้เลย 
*** Keywords ***
Verify All Products Have Name
    # ดึง element ทุกตัวที่ match ออกมาเป็น list
    @{cards}=    Get WebElements    css=.product-card .name

    # ถ้า list ว่าง loop จะไม่ทำงานเลย → test ผ่านโดยไม่ตรวจอะไร
    # ป้องกันด้วยการเช็คก่อน
    Should Not Be Empty    ${cards}    ไม่เจอ product card เลย

    FOR    ${card}    IN    @{cards}
        ${name}=    Get Text    ${card}
        Should Not Be Empty    ${name}    เจอ card ที่ไม่มีชื่อ
    END
⚠️ จุดที่พลาดบ่อย

Get WebElements ไม่ fail ถ้าไม่เจอ element — คืน list ว่างเงียบๆ ถ้าไม่เช็ค Should Not Be Empty ก่อน loop test จะผ่านทั้งที่ไม่ได้ตรวจอะไรเลย

Pattern 2

ตรวจ Table ทีละแถว

โจทย์ที่เจอบ่อย: ตาราง order, user list, report — ต้องการตรวจว่าทุกแถว status ถูกต้องหรือเปล่า

keyword ใหม่
Get Element Count locator — นับจำนวน element ที่ match คืนเป็น integer ใช้กับ IN RANGE เพื่อวนตาม index 
*** Keywords ***
All Orders Should Be Completed
    [Documentation]    ตรวจทุกแถวใน #order-table ว่า status = Completed
    ${row_count}=    Get Element Count    xpath=//table[@id='order-table']//tbody/tr
    Should Be True    ${row_count} > 0    ตารางว่างเปล่า

    # IN RANGE ไม่รองรับ expression โดยตรง ต้อง Evaluate ก่อน
    ${end}=    Evaluate    ${row_count} + 1
    FOR    ${i}    IN RANGE    1    ${end}
        ${status}=    Get Text
        ...    xpath=//table[@id='order-table']//tbody/tr[${i}]/td[3]
        Should Be Equal    ${status}    Completed
        ...    แถวที่ ${i}: status เป็น "${status}" ไม่ใช่ "Completed"
    END

XPath ที่ใช้ที่นี่คือ tr[${i}] — xpath index เริ่มที่ 1 เสมอ ซึ่งตรงกับ IN RANGE 1 ${end} พอดี

Pattern 3

กรอก Form หลาย Row จาก Data

ใน app ที่ให้ add item เป็น row เช่น invoice, shopping list, batch upload — แทนที่จะ hardcode ทีละ row ให้เอา data มา loop กรอก

*** Test Cases ***
Add Multiple Invoice Items
    Open Invoice Page
    Fill Invoice Items    แอปเปิ้ล    5    20.00
    ...                   กล้วย       3    15.00
    ...                   ส้ม         8    12.50
    Submit Invoice

*** Keywords ***
Fill Invoice Items
    [Arguments]    @{items}
    # items มาเป็น flat list: name, qty, price, name, qty, price, ...
    # ใช้ IN RANGE step=3 เพื่อจับทีละกลุ่ม
    ${total}=    Get Length    ${items}
    FOR    ${i}    IN RANGE    0    ${total}    3
        ${name}=     Set Variable    ${items}[${i}]
        ${qty}=      Set Variable    ${items}[${i+1}]
        ${price}=    Set Variable    ${items}[${i+2}]

        Click Element    css=button#add-row
        Wait Until Element Is Visible    css=.row-input:last-child

        ${row_num}=    Evaluate    ${i} // 3 + 1
        Input Text    xpath=//tbody/tr[${row_num}]/td[1]/input    ${name}
        Input Text    xpath=//tbody/tr[${row_num}]/td[2]/input    ${qty}
        Input Text    xpath=//tbody/tr[${row_num}]/td[3]/input    ${price}
    END
💡 ถ้า data ชัดเจนอยู่แล้ว

ถ้า data เป็น list of dict หรือ CSV ให้ใช้ Data-Driven Testing แทน — แต่ถ้าแค่ fill form ในระหว่าง test ไม่ต้องการ file แยก รูปแบบนี้ตอบโจทย์กว่า

Pattern 4

รอ Loading / Spinner หายก่อนทำต่อ

Sleep 5s แบบ fixed คือ anti-pattern — ถ้า load เร็วกว่านั้นก็เสียเวลาเปล่า ถ้า slow กว่านั้นก็ยัง fail อยู่ดี WHILE แก้ได้ตรงๆ

*** Keywords ***
Wait For Spinner To Disappear
    [Arguments]    ${timeout}=30 seconds
    WHILE    True    limit=${timeout}
        ${still_loading}=    Run Keyword And Return Status
        ...    Element Should Be Visible    css=.loading-spinner
        IF    not ${still_loading}    BREAK
        Sleep    0.5s
    END

Wait For Table To Load
    [Arguments]    ${table_locator}    ${timeout}=15 seconds
    WHILE    True    limit=${timeout}
        ${ready}=    Run Keyword And Return Status
        ...    Element Should Be Visible    ${table_locator}
        IF    ${ready}    BREAK
        Sleep    0.5s
    END

keyword นี้ reusable — เรียกจาก feature keyword ไหนก็ได้ แค่ส่ง locator กับ timeout ที่ต้องการ

❌ วิธีที่คนทำผิดบ่อย
Click Element    css=button#submit
Sleep    5s    # hardcode — ไม่รู้จะพอไหม
Element Should Be Visible    css=.result
✅ วิธีที่ถูก
Click Element    css=button#submit
Wait For Spinner To Disappear
Element Should Be Visible    css=.result
Pattern 5

วน Pagination จนหมดทุกหน้า

ใช้ตอนที่ต้องตรวจข้อมูลทั้งหมด หรือหา item ที่อยู่หน้าไหนก็ไม่รู้

*** Keywords ***
Find Product Across Pages
    [Arguments]    ${product_name}
    ${found}=    Set Variable    ${False}

    WHILE    True    limit=50
        # ตรวจทุก item ในหน้าปัจจุบัน
        @{items}=    Get WebElements    css=.product-list .item-name
        FOR    ${item}    IN    @{items}
            ${text}=    Get Text    ${item}
            IF    '${text}' == '${product_name}'
                ${found}=    Set Variable    ${True}
                BREAK
            END
        END

        # ถ้าเจอแล้ว ออกจาก WHILE ได้เลย
        IF    ${found}    BREAK

        # เช็คว่ามีหน้าถัดไปไหม
        ${has_next}=    Run Keyword And Return Status
        ...    Element Should Be Visible    css=button.btn-next:not([disabled])
        IF    not ${has_next}    BREAK

        Click Element    css=button.btn-next
        Wait Until Element Is Visible    css=.product-list
    END

    Should Be True    ${found}    ไม่พบ "${product_name}" ในทุกหน้า
⚠️ ตั้ง limit ไว้เสมอ

ถ้า pagination มี bug แล้วปุ่ม Next ไม่ disabled ทั้งที่หมดแล้ว WHILE จะวนไม่หยุด — limit=50 ทำให้ test fail แทนที่จะค้างค้างตลอดกาล

Pattern 6

Retry Action ที่ Flaky

บาง element เปิดมาแล้ว click ไม่ติดครั้งแรก เช่น dropdown ที่ต้อง hover ก่อน หรือ button ที่ load ช้ากว่า DOM ready แทนที่จะปล่อยให้ fail ทันที ให้ retry ก่อนสัก 3 ครั้ง

*** Keywords ***
Click With Retry
    [Arguments]    ${locator}    ${max_retry}=3
    FOR    ${attempt}    IN RANGE    ${max_retry}
        ${ok}=    Run Keyword And Return Status
        ...    Click Element    ${locator}
        IF    ${ok}    BREAK
        Sleep    1s
    END
    Should Be True    ${ok}
    ...    Click ไม่สำเร็จหลังลอง ${max_retry} ครั้ง: ${locator}

*** Test Cases ***
Submit Order With Retry
    Open Order Page
    Fill Order Form
    Click With Retry    css=button#submit-order    max_retry=3
    Wait Until Element Is Visible    css=.order-success

🧠 เช็คความเข้าใจ

ข้อ 1. Get WebElements ต่างจาก Wait Until Element Is Visible ยังไงในแง่ของการทำงาน ถ้าไม่เจอ element?

ดูเฉลย

Get WebElements ไม่ fail — คืน list ว่าง เงียบๆ

Wait Until Element Is Visible รอจนถึง timeout แล้ว fail พร้อม error message

ดังนั้นถ้าใช้ Get WebElements ต้อง Should Not Be Empty เองถ้าอยากให้ fail เมื่อไม่เจอ

ข้อ 2. ทำไมต้อง Evaluate ${row_count} + 1 แทนที่จะใส่ IN RANGE 1 ${row_count} + 1 ตรงๆ?

ดูเฉลย
FOR IN RANGE รับแค่ตัวเลขหรือ variable เดี่ยว ไม่ evaluate expression — ต้องคำนวณให้เสร็จก่อนด้วย Evaluate แล้วค่อยส่ง variable เข้าไป

ข้อ 3. Pattern WHILE + Spinner กับ Wait Until Element Is Not Visible ต่างกันยังไง? แบบไหนควรใช้เมื่อไหร่?

ดูเฉลย

Wait Until Element Is Not Visible ง่ายกว่า ใช้เมื่อต้องการแค่รอ element หายไป — ควรใช้ก่อน

WHILE เหมาะเมื่อต้องทำอะไรบางอย่างระหว่างรอด้วย เช่น log สถานะแต่ละรอบ, เช็คหลาย condition พร้อมกัน, หรือ fallback เมื่อ timeout
