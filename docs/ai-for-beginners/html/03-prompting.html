ศิลปะการ Prompt

⏱ 30 นาที 🔥 สำคัญมาก 🛠 ใช้งานได้ทันที ✅ มี Interactive

🔄 ก่อนอ่านบทนี้ — ทบทวนจากบทที่ 2

ลองตอบก่อนดูเฉลย:

"LLM สร้าง output อย่างไร? มันค้นหาคำตอบจาก database หรือสร้างทีละ token?"

ดูเฉลย
LLM สร้าง output ทีละ token แบบ autoregressive — มันทำนายว่า token ต่อไปที่น่าจะมาคืออะไร โดยอ้างอิงจาก context ทั้งหมดก่อนหน้า ไม่ได้ "ค้นหา" คำตอบจากที่ไหน แต่ใช้ pattern ที่เรียนมาจาก training data เพื่อ generate คำต่อคำ

🎯 อ่านจบบทนี้แล้วคุณจะทำได้

  • อธิบายว่า Prompt ที่ดีต่างจาก Prompt แย่อย่างไร และทำไมถึงสำคัญ
  • ใช้ RCTFC Framework (Role, Context, Task, Format, Constraints) เขียน Prompt อย่างเป็นระบบ
  • รู้จักเทคนิค Few-shot และ Chain-of-Thought และเลือกใช้ได้ถูกสถานการณ์
  • หลีกเลี่ยง Common Mistakes ที่ทำให้ได้ผลลัพธ์แย่กว่าที่ควร
  • เขียน Prompt สำหรับงานจริงในชีวิตได้ทันที
💬 ทำไมต้องรู้?

คุณเคยถาม AI แล้วได้คำตอบที่ "พอใช้ได้" แต่รู้สึกว่ามันยังไม่ตรงกับที่ต้องการ? หรือบางทีตอบยาวเกินไป บางทีสั้นเกินไป บางทีผิด context ไปเลย?

ปัญหาส่วนใหญ่ไม่ได้อยู่ที่ AI — แต่อยู่ที่วิธีที่เราสื่อสารกับมัน AI ไม่รู้ว่าคุณมี background อะไร ต้องการ format แบบไหน มีข้อจำกัดอะไร ถ้าคุณไม่บอก มันก็ต้อง "เดา" — และการเดาก็ผิดบ้างถูกบ้าง

Prompting ที่ดีคือทักษะ ไม่ใช่โชค — และทักษะนี้เรียนรู้ได้ใน 30 นาที

💡 Analogy

เหมือนการ Briefing งานพนักงานใหม่ที่ฉลาดมาก

ลองนึกถึงสถานการณ์นี้ — คุณจ้างพนักงานใหม่ที่ฉลาดมาก มีความรู้กว้าง แต่ยังไม่รู้อะไรเกี่ยวกับงานของคุณเลย

ถ้าคุณบอกแค่ว่า "ช่วยทำรายงานให้หน่อย" — พนักงานก็ต้องเดาทุกอย่าง: รายงานเรื่องอะไร? ยาวแค่ไหน? format แบบไหน? audience คือใคร? deadline เมื่อไร?

แต่ถ้าคุณบอกว่า "คุณเป็นนักวิเคราะห์ตลาด ช่วยสรุปผลขายเดือนมีนาคม ตอบเป็น bullet 5 ข้อ สำหรับประชุมผู้บริหาร อย่าใส่ตัวเลขดิบ ให้ใส่เปอร์เซ็นต์เปลี่ยนแปลงแทน" — คุณจะได้งานที่ตรงมากกว่า

Prompt คือ briefing ของคุณ ยิ่งชัด ยิ่งได้งานตรงความต้องการ

⚠️ ถ้าเชื่อ analogy นี้ 100% จะเข้าใจผิดว่า:
  • AI "รู้สึก" เหมือนพนักงานจริงๆ — AI ไม่มีอารมณ์ ไม่รู้สึกเบื่อ ไม่รู้สึกดีใจ และไม่มีแรงจูงใจ ตอบดีเพราะสถิติ ไม่ใช่เพราะ "ตั้งใจ"
  • การ "สุภาพ" กับ AI ส่งผลเหมือนกับคน — การพูด "ขอบคุณ" หรือ "please" ส่งผลบ้างในบางกรณี แต่เป็นเพราะ training data มีรูปแบบนั้น ไม่ใช่เพราะ AI รู้สึกดีขึ้น
  • AI "จำ" สิ่งที่คุณบอกตลอดไป — พนักงานจริงจดจำ context ข้ามสัปดาห์ได้ แต่ AI ลืมทุกอย่างเมื่อ session จบ (ยกเว้นจะมีระบบ memory พิเศษ)

📐 RCTFC Framework — โครงสร้าง Prompt ที่ได้ผล

จากการศึกษา Prompt Engineering จาก Anthropic และ OpenAI พบว่า Prompt ที่ได้ผลดีมักประกอบด้วยส่วนต่างๆ เหล่านี้ — เราสรุปเป็น RCTFC Framework ให้จำง่าย

🎭
R — Role
บอกให้ Claude เล่นบทบาทอะไร เช่น "คุณเป็นนักโภชนาการที่มีประสบการณ์ 10 ปี" — ช่วยให้ตอบในมุมที่ถูกต้อง
📋
C — Context
ให้ข้อมูลพื้นหลังที่จำเป็น เช่น "ฉันอายุ 30 ปี น้ำหนัก 70 กก." — AI จะได้ไม่ต้องเดาสิ่งที่คุณรู้อยู่แล้ว
🎯
T — Task
บอกชัดว่าต้องการอะไร เช่น "แนะนำแผนอาหารสำหรับ 1 สัปดาห์" — Task ที่คลุมเครือ = ผลลัพธ์ที่คลุมเครือ
📐
F — Format
ระบุรูปแบบ output เช่น "ตอบเป็นตาราง" หรือ "bullet 5 ข้อ ไม่เกิน 2 ประโยค/ข้อ" — ไม่งั้น AI เลือก format เอง
⚠️
C — Constraints
ข้อจำกัดที่ต้องรู้ เช่น "ไม่กินเนื้อสัตว์" หรือ "งบ 300 บาท/วัน" — ป้องกันไม่ให้ได้คำแนะนำที่ใช้จริงไม่ได้
ℹ️
ไม่จำเป็นต้องใช้ครบทุกส่วนทุกครั้ง — ถามง่ายๆ เช่น "ราชธานีของฝรั่งเศสคืออะไร" ไม่ต้องใส่ Role หรือ Format ก็ได้ใช้ RCTFC เมื่อ task ซับซ้อนหรือต้องการความแม่นยำสูง

ลองดู Prompt Builder แบบ Interactive

กด "เปิดเผยส่วนถัดไป" เพื่อดูว่าแต่ละส่วนของ RCTFC เพิ่ม value อะไรให้ Prompt

🔧 เทคนิค Prompting ขั้นสูง

1. Few-shot Prompting — สอนด้วยตัวอย่าง

แทนที่จะอธิบาย task ยาวๆ ให้แสดงตัวอย่าง input→output 2-3 คู่ก่อน แล้วถามสิ่งที่ต้องการจริง

📌 เปรียบเทียบ: บอก vs แสดง
วิธี ตัวอย่าง Prompt เมื่อใช้ได้ผลดี
บอก (Tell) แปลประโยคต่อไปนี้เป็นภาษาไทยแบบสุภาพ formal Task ง่าย, คำว่า "formal" เข้าใจตรงกัน
แสดง (Few-shot) Input: "Thanks!" → Output: "ขอบคุณมากครับ"
Input: "Sorry" → Output: "ขออภัยเป็นอย่างยิ่งครับ"
Input: "See you!" → ?		 Task มี style เฉพาะที่อธิบายยาก หรือต้องการ consistency สูง
💡
Few-shot ดีที่สุดเมื่อ: คุณต้องการรักษา tone, format หรือ style เฉพาะที่อธิบายด้วยคำพูดได้ยาก เช่น "เขียนแบบ casual แต่ professional" — แทนที่จะอธิบายให้ยาว ให้ตัวอย่าง 2-3 อันแทน

2. Chain-of-Thought — ให้ Claude "คิดออกเสียง"

สำหรับ task ที่ต้องการ reasoning ซับซ้อน เช่น คณิตศาสตร์, การวิเคราะห์, การตัดสินใจ — ให้ Claude แสดงขั้นตอนการคิดก่อนตอบ

❌ แบบที่ไม่ดี
ถ้า Peter มีแอปเปิ้ล 3 ผล แล้วให้ Mary 1 ผล แล้ว Mary ให้ Tom 2 ผล Tom มีกี่ผล?

→ AI อาจตอบสั้นๆ และผิดได้ง่าย

✅ Chain-of-Thought
ถ้า Peter มีแอปเปิ้ล 3 ผล แล้วให้ Mary 1 ผล แล้ว Mary ให้ Tom 2 ผล Tom มีกี่ผล?

คิดทีละขั้นตอนก่อนตอบ

→ AI จะ trace ตัวเลขแต่ละขั้น แม่นยำกว่า

วลีที่ใช้ได้: "คิดทีละขั้นตอนก่อน", "Let's think step by step", "อธิบายกระบวนการคิดก่อนสรุป"

3. XML Tags — แยก sections ให้ชัดเจน

เมื่อ Prompt มีหลายส่วน เช่น context ยาวๆ + task + ตัวอย่าง ใช้ XML tags แยกให้ชัด Claude อ่าน structure ได้ดีกว่า

<context>
บริษัทของเราขายซอฟต์แวร์ HR ให้ SME ในไทย มีลูกค้า 200 ราย
ปัญหาหลักที่ลูกค้าร้องเรียนคือ UI ซับซ้อนเกินไป
</context>

<task>
เขียน email ถึงลูกค้าเพื่อแนะนำ feature ใหม่ที่ทำให้ใช้งานง่ายขึ้น
</task>

<format>
- ความยาว 150-200 คำ
- เป็นทางการ แต่ friendly
- มี call-to-action ให้จองเวลา demo
</format>
📌
ทำไม XML tags ถึงดี? Anthropic แนะนำการใช้ XML tags โดยเฉพาะกับ Claude เพราะ Claude ถูก train ให้เข้าใจ structure นี้ดีเป็นพิเศษ ช่วยลด ambiguity เมื่อ prompt มีหลาย component

ดูตัวอย่าง Prompt แย่ vs ดี

📝 ตัวอย่าง 3 ระดับ

Beginner

สรุปบทความข่าวอย่างมีระบบ

สถานการณ์: คุณเพิ่งอ่านบทความยาว 2,000 คำเกี่ยวกับนโยบายเศรษฐกิจใหม่ และต้องการสรุปเพื่อส่งให้เพื่อนร่วมทีม

❌ Prompt แบบ Beginner ทั่วไป
สรุปบทความนี้ให้หน่อย: [วางเนื้อหาบทความ]

→ ได้สรุปมาอาจยาวมาก อาจสั้นเกินไป อาจเน้นส่วนที่ไม่ใช่ priority

✅ Prompt ที่ดีกว่า (ใช้ T + F + C)
สรุปบทความต่อไปนี้โดย:
- เขียนเป็น bullet 5 ข้อ ข้อละไม่เกิน 1 ประโยค
- เน้นผลกระทบต่อธุรกิจ SME เป็นหลัก
- ใช้ภาษาเข้าใจง่าย ไม่ใช้ jargon เศรษฐศาสตร์

[วางเนื้อหาบทความ]

→ ได้สรุป 5 ข้อ เน้นมุมที่ต้องการ ใช้ภาษาที่ส่งต่อให้ทีมได้ทันที

💡
Lesson: แม้แค่ใส่ Format (bullet 5 ข้อ) + Context (SME) + Constraint (ไม่ใช้ jargon) ก็ทำให้ผลลัพธ์ดีขึ้นมาก ไม่ต้องใช้ RCTFC ครบทุกส่วนเสมอไป
Intermediate

เขียน Cover Letter สำหรับสมัครงาน

สถานการณ์: คุณสมัครตำแหน่ง UX Designer ที่ fintech startup และต้องการ cover letter ที่โดดเด่น

❌ Prompt ที่พลาด
ช่วยเขียน cover letter สำหรับ UX Designer ให้หน่อย

→ ได้ generic template ที่ใช้ได้กับทุก position ซึ่งแปลว่าไม่ได้เฉพาะเจาะจงสำหรับ job นี้เลย

✅ Prompt ครบ RCTFC
<role>คุณเป็น career coach ที่เชี่ยวชาญ tech industry</role>

<context>
ผู้สมัคร: มีประสบการณ์ UX 3 ปี เน้น mobile app
ทำโปรเจค: redesign แอปธนาคาร ลด support ticket 40%
ตำแหน่งที่สมัคร: Senior UX Designer ที่ MoneyTech (fintech startup)
Job description เน้น: user research, data-driven design, collaboration กับ engineering
</context>

<task>
เขียน cover letter ที่เชื่อมประสบการณ์ของผู้สมัครเข้ากับความต้องการของ MoneyTech
โดยเน้น fintech experience และ impact ที่วัดผลได้
</task>

<format>
- ความยาว 200-250 คำ
- มี 3 ย่อหน้า: opening hook, evidence, closing CTA
- ไม่เริ่มด้วย "ฉันสนใจ..." หรือ "I am writing to..."
</format>

<constraints>
- ไม่พูดเกินจริง — ใช้เฉพาะข้อมูลที่ให้ไว้
- ภาษาอังกฤษ formal แต่ไม่ robotic
</constraints>

→ ได้ cover letter ที่ personalized, มี structure ชัด, ใช้ตัวเลขจริง — ต่างจาก generic template ชัดเจน

⚡
Pro tip: ใส่ข้อมูล เฉพาะของคุณ มากที่สุดเท่าที่มี ตัวเลข (40% reduction) และชื่อบริษัทจริง ทำให้ output ต่างจาก template ชัดเจน
Advanced

Data Analysis ด้วย Few-shot + Chain-of-Thought

สถานการณ์: คุณมีข้อมูลยอดขายและต้องการให้ Claude วิเคราะห์ trend พร้อม give actionable insights — ไม่ใช่แค่ describe ตัวเลข

✅ Few-shot + Chain-of-Thought Prompt
<role>
คุณเป็น Senior Business Analyst ที่เชี่ยวชาญ retail analytics
สไตล์การวิเคราะห์: ดูข้อมูล → หา pattern → เสนอ hypothesis → suggest action
</role>

<few-shot-examples>
ตัวอย่างที่ 1:
ข้อมูล: ยอดขาย Jan=100K, Feb=95K, Mar=140K
วิเคราะห์: Feb ต่ำกว่า Jan 5% (seasonal dip หลัง New Year) แต่ Mar กระโดด 47%
→ Hypothesis: มี promotion หรือ event ใน Mar
→ Action: ตรวจสอบ campaign ที่ run ใน Mar เพื่อ replicate

ตัวอย่างที่ 2:
ข้อมูล: Product A=500 units, Product B=50 units, ราคา A=100B, B=500B
วิเคราะห์: B มี revenue/unit สูงกว่า 5x แต่ volume ต่ำ
→ Hypothesis: B อาจมี market นิ่ม หรือ awareness ต่ำ
→ Action: ทำ customer survey ว่าทำไมไม่ซื้อ B
</few-shot-examples>

<task>
วิเคราะห์ข้อมูลต่อไปนี้ในสไตล์เดียวกัน โดยคิดทีละขั้นตอนก่อนสรุป:

ยอดขายรายเดือน (บาท):
- Jan: 2,400,000
- Feb: 2,100,000
- Mar: 1,900,000
- Apr: 3,200,000
- May: 3,500,000

Top 3 สินค้า:
- สินค้า X: 1,200 units @ 500B = 600,000B
- สินค้า Y: 200 units @ 2,000B = 400,000B
- สินค้า Z: 800 units @ 300B = 240,000B
</task>

<format>
1. Pattern ที่เห็น (bullet)
2. Hypotheses สำหรับแต่ละ pattern
3. Recommended actions (prioritize by impact)
ไม่เกิน 350 คำ
</format>
🔬
ทำไม combination นี้ถึงทรงพลัง:
  • Few-shot: สอน style การวิเคราะห์ที่ต้องการด้วยตัวอย่างจริง แทนการอธิบาย
  • Chain-of-Thought: ขอให้คิดทีละขั้น ลด hallucination และได้ reasoning ที่ trace ได้
  • XML Tags: แยก examples กับ task จริงชัดเจน AI ไม่สับสน
  • Role + Format: จำกัด scope ให้ตอบในสไตล์ที่ต้องการ ไม่ออกนอกเรื่อง

⚠️ Common Mistakes

ข้อผิดพลาดเหล่านี้มาจาก Anthropic Prompt Engineering documentation — เป็นปัญหาที่พบบ่อยที่สุด

❌ "สรุปเรื่องนี้"
✅ "สรุปเป็น bullet 5 ข้อ แต่ละข้อไม่เกิน 2 ประโยค เน้นผลกระทบต่อธุรกิจ"
ไม่ระบุ format ทำให้ AI เลือก format เอง — อาจได้ paragraph ยาว 10 บรรทัด ทั้งที่ต้องการ bullet สั้นๆ
*(source: Anthropic prompt engineering docs)*
❌ Prompt ยาวเป็นพรืด ไม่มีโครงสร้าง
✅ ใช้ XML tags (<context>, <task>, <format>) แยก sections ให้ชัดเจน
Prompt ที่ไม่มี structure ทำให้ AI สับสนว่าอะไรคือ context อะไรคือ instruction อะไรคือ constraint — โดยเฉพาะเมื่อ prompt ยาวกว่า 200 คำ
*(source: Anthropic prompt engineering docs)*
❌ ไม่บอก context ที่ AI ต้องรู้ก่อน ถามตรงๆ เลย
✅ ให้ข้อมูลพื้นหลังที่จำเป็นก่อนเสมอ เช่น ใคร? สถานการณ์อะไร? ต้องการใช้ทำอะไร?
AI ไม่รู้อะไรเกี่ยวกับ context ของคุณนอกจากที่คุณบอก ถ้าไม่บอก มันต้องเดา — และการเดาทำให้ได้ generic answer ที่ไม่ตรงสถานการณ์

❌ ถามหลายคำถามพร้อมกันใน 1 message รวมกันเป็นย่อหน้า
✅ แยกถามทีละเรื่อง หรือ numbering ชัดเจน เช่น "ถามสองเรื่อง: 1) ... 2) ..."
AI มักตอบคำถามแรกได้ดี แต่ตอบคำถามหลังๆ สั้นลงเรื่อยๆ หรือลืมบางข้อไปเลย การ numbering ชัดทำให้ AI ครบทุกข้อ

📚 สรุปบทที่ 3

  • RCTFC Framework — Role, Context, Task, Format, Constraints คือโครงสร้างพื้นฐานของ Prompt ที่ดี ไม่ต้องใช้ครบทุกครั้ง แต่รู้ว่าแต่ละส่วนช่วยอะไร
  • Few-shot ดีกว่าการอธิบายยาวๆ เมื่อต้องการรักษา style เฉพาะ — สอนด้วยตัวอย่างดีกว่าบอกด้วยคำพูด
  • Chain-of-Thought — เพิ่ม "คิดทีละขั้นตอนก่อน" เมื่อ task ต้องการ reasoning ซับซ้อน ช่วยลด error
  • XML Tags — แยก context / task / format ด้วย tags เมื่อ prompt มีหลาย component ยาวกว่า 200 คำ
  • Format ขาดไม่ได้ — ถ้าไม่ระบุ format AI จะเลือกเอง และอาจไม่ตรงกับที่ต้องการ

คำถามที่ 1

"ลองเขียน Prompt ขอให้ Claude ช่วยเขียน Email ขอโทษลูกค้า กรณีระบบ payment ล่มทำให้ transaction ไม่สำเร็จ 2 ชั่วโมง โดยใช้ RCTFC framework ครบทุกส่วน"

ดูตัวอย่าง Prompt ที่ดี
R: คุณเป็น Customer Success Manager ที่เชี่ยวชาญการสื่อสารในสถานการณ์ sensitive
C: ระบบ payment ของบริษัทล่มระหว่าง 14:00-16:00 น. ทำให้ลูกค้าทำ transaction ไม่ได้ โดยลูกค้าที่ได้รับผลกระทบคือผู้ใช้ premium 500 คน
T: เขียน email ขอโทษและอธิบายสิ่งที่เกิดขึ้น พร้อมระบุว่าจะป้องกันอย่างไรในอนาคต
F: ความยาว 150-200 คำ มีหัวข้อ Subject line ด้วย เป็น 3 ย่อหน้า

คำถามที่ 2

"Chain-of-Thought prompting ช่วยได้ในสถานการณ์แบบไหน? ยกตัวอย่างสถานการณ์จากชีวิตจริงที่ไม่ได้อยู่ในบทเรียนนี้"

ดูแนวคำตอบ
Chain-of-Thought ดีสำหรับ: งานที่ต้องการ multi-step reasoning เช่น การวิเคราะห์ legal document (ต้อง apply กฎหมายหลายมาตรา), การวางแผนงบประมาณ (ต้องคำนวณทีละขั้น), การ debug code (ต้อง trace logic), หรือการตัดสินใจเลือก vendor (ต้องชั่งน้ำหนักหลาย criteria) — โดยทั่วไปคือ task ที่ถ้าตอบ "ตรงๆ" โดยไม่แสดง reasoning มีโอกาสผิดสูง
คำถามที่ 3

"Few-shot examples แตกต่างจากการอธิบาย task อย่างไร? อันไหนดีกว่าในสถานการณ์ใด?"

ดูแนวคำตอบ
การอธิบาย task: ดีกว่าเมื่อ requirement ชัดเจน สื่อสารด้วยคำพูดได้ง่าย เช่น "แปลเป็น formal Thai" หรือ "สรุป 5 ข้อ"

Few-shot examples: ดีกว่าเมื่อ style ที่ต้องการ "อธิบายยาก แต่ดูแล้วรู้" — เช่น tone ของ brand, สไตล์การเขียน copy ที่เฉพาะ, หรือ format output ที่ซับซ้อน ตัวอย่าง 2-3 อันบอกได้มากกว่าคำอธิบาย 1 หน้ากระดาษ
