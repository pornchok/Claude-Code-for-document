⚡ บทที่ 4

Claude Superpowers

เมื่อ Claude ธรรมดากลายเป็น Claude ที่ทำงานตาม workflow ของผู้เชี่ยวชาญ — เข้าใจ Skills System และใช้มันให้ถูกวิธี

⏱ 20 นาที ⚡ Skills System 🎯 Priority Rules 🔄 Workflow-driven

🔄 ก่อนอ่านบทนี้ — ทบทวนจากบทที่ 3

ลองตอบก่อนดูเฉลย — ถ้าจำได้ไม่หมด ไม่เป็นไร แค่ลองก่อน:

"RCTFC framework ย่อมาจากอะไร และแต่ละส่วนช่วยอะไร?"

ดูเฉลย
Role (บทบาทของ AI) — บอกว่าให้ Claude เป็น "ใคร" เพื่อปรับโทนและความเชี่ยวชาญ
Context (ข้อมูลพื้นหลัง) — ให้ข้อมูลที่จำเป็นสำหรับการตอบ
Task (สิ่งที่ต้องการ) — ระบุให้ชัดว่าต้องการอะไร
Format (รูปแบบ output) — บอกว่าต้องการผลลัพธ์ในรูปแบบใด
Constraints (ข้อจำกัด) — สิ่งที่ไม่ต้องการหรือข้อแม้ต่างๆ

ทั้งห้าส่วนทำให้ prompt ชัดเจนและได้ผลลัพธ์ที่ตรงกว่า เพราะลด ambiguity ในแต่ละมิติ

🎯 อ่านจบบทนี้แล้วคุณจะทำได้

  • อธิบายได้ว่า Claude Superpowers / Skills System คืออะไร และต่างจาก Claude default อย่างไร
  • เข้าใจ priority order: User > Skills > Default และรู้ว่า "ใครชนะใคร" ในสถานการณ์ต่างๆ
  • บอกได้ว่ามี Skills อะไรบ้าง และแต่ละ Skill เหมาะกับงานประเภทใด
  • invoke Skills ได้อย่างถูกต้อง ทั้งแบบพิมพ์คำสั่งเองและแบบที่ Claude detect เอง
  • อธิบายได้ว่าทำไม Skills ถึงดีกว่าการเขียน Prompt ยาวๆ สำหรับบางงาน
🤔 ทำไมต้องรู้?

Claude default นั้น "ฉลาด" อยู่แล้ว — ถามอะไรก็ตอบได้ ช่วยงานได้หลากหลาย แต่ปัญหาคือ Claude default improvise ทุกครั้ง ไม่มี workflow ที่ตายตัว ผลลัพธ์จึงขึ้นอยู่กับว่า Claude "เดาถูก" ว่าคุณต้องการอะไรมากน้อยแค่ไหน

Skills เปลี่ยน HOW Claude ทำงาน ทั้งกระบวนการ — ไม่ใช่แค่ "บอกข้อมูลเพิ่ม" เหมือนความแตกต่างระหว่าง "ใช้ word processor ธรรมดา" กับ "ใช้ word processor ที่มี extension เฉพาะงาน" ตัวแรกทำได้ทุกอย่างแบบ generic ตัวหลังทำงานนั้นได้ดีกว่าเพราะออกแบบมาเพื่อมัน

ถ้าคุณใช้ Claude Code แต่ไม่รู้ว่า Skills คืออะไร คุณอาจพลาดโอกาสที่จะได้ผลลัพธ์ ที่ดีกว่าเดิมมากโดยไม่ต้องเขียน Prompt ซับซ้อนขึ้นเลย

💡 Analogy — Claude เป็นเหมือนอะไร?

Analogy

Claude เปรียบเหมือน smartphone ที่ฉลาดมาก — มันทำได้หลายอย่างออก-of-the-box จะโทร ส่ง SMS ถ่ายรูป ก็ทำได้หมด นั่นคือ "Claude default"

แต่พอคุณ install Apps เฉพาะทาง — เช่น แอปตกแต่งรูปของ pro photographer หรือ แอป GPS สำหรับนักวิ่ง marathon — มันเริ่มทำงานตาม workflow ที่ออกแบบมาเพื่องานนั้น โดยเฉพาะ มีขั้นตอน มีเครื่องมือ มี checklist ที่ผู้เชี่ยวชาญสร้างไว้ให้แล้ว

Skills ก็เหมือน Apps เหล่านั้น — install เมื่อไร Claude ก็เปลี่ยน จาก "ทำทุกอย่างแบบ generic" เป็น "ทำงานนี้ตาม proven workflow"

⚠️ ถ้าเชื่อ Analogy นี้ 100% จะเข้าใจผิดว่า:
  • Skills ทำงาน "แยก" จาก Claude — จริงๆ แล้ว Skills คือ instruction text ที่ Claude อ่านแล้ว follow เหมือน Apps บน smartphone ที่รันแยก แต่ Skills ไม่ใช่ แค่เป็นคำแนะนำที่ Claude นำไปประมวลผลในหัวตัวเอง
  • Skills ใช้ "compute" มากกว่า — จริงๆ Skills ใช้ tokens มากกว่า (เพราะต้องโหลด instruction text) ไม่ใช่ compute หรือ processing power ที่มากกว่า ความแตกต่างอยู่ที่ context window ที่ใช้ ไม่ใช่ hardware
  • Skills active ตลอดเวลาเหมือน App ที่ install แล้ว — จริงๆ แล้ว Skills ต้องถูก invoke ก่อนจึงจะทำงาน ไม่ใช่ว่า install แล้วทำงานอัตโนมัติทุกครั้ง เหมือน App ที่ต้องเปิดก่อนใช้ ถ้าไม่ invoke Skill Claude จะทำงาน default ตามปกติ

📚 เนื้อหาหลัก

1. Claude Superpowers / Skills คืออะไร?

Skills System คือชุดของ "คำแนะนำพิเศษ" ที่บอก Claude ว่าต้องทำงาน อย่างไรในบริบทนั้นๆ ลองนึกภาพว่าคุณจ้างที่ปรึกษา — ถ้าคุณบอกแค่ "ช่วยดูโค้ดให้หน่อย" เขาจะดูแบบ generic แต่ถ้าคุณบอก "ดูโค้ดตาม security checklist ของ OWASP" เขาจะดูตาม framework ที่ชัดเจน

เมื่อ invoke Skill Claude จะ:

  • โหลด instructions จาก Skill file (ข้อความที่บอก workflow)
  • ทำงานตาม workflow ที่กำหนดไว้ใน Skill นั้น
  • ใช้ tools ตามที่ Skill อนุญาตและแนะนำ
ℹ️
Skills ไม่ใช่ magic — มันเป็น text instructions ที่คนออกแบบระบบเขียนไว้ล่วงหน้า เหมือน SOP (Standard Operating Procedure) ที่พนักงานต้องทำตาม ความแตกต่างจาก Prompt ธรรมดา คือ Skills ถูกออกแบบโดย ผู้เชี่ยวชาญ และทดสอบมาแล้วว่าได้ผล

2. Priority Order — ใครชนะใคร?

นี่คือสิ่งที่สำคัญที่สุดในบทนี้ เมื่อมีคำสั่งขัดแย้งกัน Claude ใช้ priority นี้ตัดสิน:

1. User Instructions
(CLAUDE.md, direct requests, คำสั่งในบทสนทนา) ← สูงสุด — เสมอ
2. Superpowers Skills
(Skill instructions ที่ invoke) ← override default behavior
3. Default System Prompt
(behavior พื้นฐานของ Claude) ← ต่ำสุด

ตัวอย่างที่ชัดที่สุด: ถ้าคุณบอกว่า "อย่าใช้ TDD" ในไฟล์ CLAUDE.md แต่ Skill ที่ invoke อยู่บอกว่า "ต้องเขียน test ก่อนเสมอ" — Claude จะทำตาม คุณ ไม่ใช่ตาม Skill เพราะ User > Skill

⚠️
ระวังความเข้าใจผิด: บางคนคิดว่า Skill มี "authority" มากกว่า user เพราะ Skill ถูกออกแบบโดย expert — แต่ไม่ใช่ คุณในฐานะ User มีสิทธิ์สูงสุดเสมอ Skills เป็นแค่ "แนวทางแนะนำ" ที่คุณสามารถ override ได้ตลอดเวลา

3. Skills ที่มีให้ใช้ (และใช้เมื่อไร)

นี่คือ Skills หลักๆ ที่มักใช้บ่อย พร้อมคำแนะนำว่าควร invoke เมื่อไร:

Skill ใช้เมื่อ สิ่งที่เปลี่ยน
brainstorming ก่อนสร้าง feature ใหม่ หรือก่อนตัดสินใจ approach ใดๆ Claude จะสำรวจ intent, requirements, และ design ก่อนลงมือทำ
test-driven-development ก่อนเขียน code ใดๆ (ไม่ใช่หลังเขียน) Claude จะเขียน test ก่อน แล้วค่อย implement ให้ test ผ่าน
systematic-debugging เจอ bug ที่หาสาเหตุไม่เจอ หรือ bug ที่ซับซ้อน Claude จะวิเคราะห์ systematically ก่อน propose fix — ไม่ใช่ guess
writing-plans มี spec หรือ requirements แล้ว ก่อนลงมือเขียน code Claude จะสร้าง implementation plan ที่ review ได้ก่อนทำจริง
verification-before-completion ก่อนบอกว่างานเสร็จ หรือก่อน commit/push Claude จะ verify ผลลัพธ์จริงก่อน claim ว่า "เสร็จแล้ว"
code-review ตรวจ code ก่อน merge หรือ PR review Claude จะ review อย่างเป็นระบบตาม checklist ที่ครอบคลุม
💡
ไม่มี Skill ที่ "ดีที่สุด" เสมอ — แต่ละ Skill ดีที่สุดในบริบทของตัวเอง การเลือก Skill ผิดไม่ทำให้ Claude ทำงานผิด แต่อาจทำให้ได้ผลลัพธ์ที่ไม่ optimized สำหรับงานนั้น

4. วิธี Invoke Skills

มีสองทาง:

⌨️
พิมพ์ /skill-name เอง
ผู้ใช้พิมพ์ /brainstorming หรือ /systematic-debugging ตรงๆ ใน prompt — Claude Code จะ recognize และ invoke skill นั้นทันที ใช้เมื่อคุณรู้ว่าต้องการ Skill ใดชัดเจน
🤖
Claude detect เอง
ถ้า task ที่คุณส่งมา match กับ Skill ใด Claude จะ invoke เอง ตามกฎ: ถ้ามีโอกาส 1% ขึ้นไปที่ Skill จะเกี่ยวข้อง ต้อง invoke เสมอ (conservative approach เพื่อไม่พลาด workflow ที่จำเป็น)

5. ทำไม Skills ดีกว่าแค่เขียน Prompt ยาวๆ?

คำถามที่หลายคนถาม: "แทนที่จะใช้ Skill ฉันเขียน prompt ยาวๆ อธิบาย workflow เองไม่ได้เหรอ?" คำตอบคือ ได้ — แต่ได้ผลน้อยกว่า เพราะ:

  • Expert design — Skills ถูกออกแบบโดยคนที่รู้ว่า workflow ไหนให้ผลดีที่สุด คุณอาจเขียน prompt ที่คิดว่าครอบคลุมแล้ว แต่พลาด edge case ที่ expert รู้
  • Consistency — Skill เดิมให้ workflow เดิมทุกครั้ง Prompt ที่เขียนเองอาจ "ตีความ" ต่างกันในแต่ละครั้ง
  • ประหยัดเวลา — ไม่ต้อง re-explain workflow ทุกครั้ง แค่ invoke Skill
  • Proven results — Skills ผ่านการทดสอบมาแล้วในสถานการณ์จริง ไม่ใช่ improvise

🧪 ตัวอย่าง 3 ระดับ

Beginner — ความแตกต่างที่มองเห็นได้

ลอง invoke /brainstorming ก่อนสร้าง feature ใหม่

สมมุติคุณต้องการเพิ่ม "ระบบแจ้งเตือนเมื่อ stock เหลือน้อย" ใน e-commerce app ลองเปรียบเทียบสองแบบ:

แบบที่ 1 — ไม่ใช้ Skill (Claude default):

คุณ: ช่วยเพิ่มระบบแจ้งเตือนเมื่อ stock เหลือน้อยให้หน่อย

Claude: โอเค ฉันจะสร้างระบบแจ้งเตือนที่ตรวจ stock ทุก 1 ชั่วโมง
        และส่ง email เมื่อ stock < 10 ชิ้น มาเริ่มเขียน code กัน...

Claude ตรงไปสร้าง code เลย — แต่ไม่รู้ว่าคุณต้องการ threshold เท่าไร, แจ้งใคร, แจ้งผ่านช่องทางอะไร ฯลฯ

แบบที่ 2 — ใช้ /brainstorming Skill:

คุณ: /brainstorming ช่วยเพิ่มระบบแจ้งเตือนเมื่อ stock เหลือน้อย

Claude: ก่อนลงมือสร้าง ขอสำรวจ requirements ก่อนนะ:

        1. ต้องการใครบ้างที่ได้รับการแจ้งเตือน? (admin, warehouse manager, ทีม ops?)
        2. threshold "เหลือน้อย" คือเท่าไร? แต่ละ product มี threshold ต่างกันได้ไหม?
        3. ช่องทางแจ้งเตือน: email, LINE, Slack, หรือ in-app notification?
        4. ต้องการ escalation ไหม? ถ้าไม่มีใคร acknowledge ภายใน X ชั่วโมง?
        5. แจ้งทันทีที่ stock ลดถึง threshold หรือแจ้งตามรอบ (hourly, daily)?

        ช่วยตอบก่อน แล้วเราจะ design solution ที่ตรงกับความต้องการจริงๆ

ความแตกต่างชัดมาก — แบบที่ 2 ตั้งคำถามก่อนสร้าง เพราะ brainstorming Skill บอก Claude ว่าต้องสำรวจ intent และ requirements ก่อนลงมือ

Intermediate — workflow ที่ structured

ใช้ /systematic-debugging กับ bug ที่หาสาเหตุไม่เจอ

คุณพัฒนา API สำหรับระบบ booking โรงแรม และพบว่า endpoint POST /reservations บางครั้ง return 500 Internal Server Error แต่ไม่ใช่ทุก request error เกิดประมาณ 1 ใน 10 ครั้ง และ reproduce ไม่ได้ทุกครั้ง 

คุณ: /systematic-debugging
      POST /reservations คืน 500 แบบ intermittent ประมาณ 10% ของ requests
      reproduce ไม่ได้ทุกครั้ง ไม่รู้ว่าเกิดจากอะไร

Claude: [Systematic Debugging Skill activated]

        ขอ information เพิ่มก่อน:

        PHASE 1 — Understand the symptom:
        - error message ใน log บอกว่าอะไร? (full stack trace)
        - 500 เกิดในเวลาที่เฉพาะเจาะจง (peak hours) หรือ random?
        - เกิดเฉพาะ request บางประเภทไหม? (room type, date range, payment method?)
        - เริ่มเกิดตอนไหน? มี deployment ก่อนหน้านั้นไหม?

        PHASE 2 — Form hypotheses (จากข้อมูลที่มี):
        hypothesis A: Race condition ใน concurrent booking (เช่น double-booking check)
        hypothesis B: Database connection pool exhaustion
        hypothesis C: Third-party payment API timeout ที่ไม่ได้ handle

        PHASE 3 — Test plan (เรียงตาม probability):
        1. ดู log ใน production ช่วง 24 ชั่วโมงที่ผ่านมา
        2. เพิ่ม structured logging รอบ database calls
        3. Monitor connection pool stats
        4. Reproduce ใน staging ด้วย concurrent load test

        ส่ง stack trace มาให้ดูหน่อยได้เลย เราจะเจาะ hypothesis ที่น่าสงสัยที่สุดก่อน

ถ้าไม่ใช้ Skill Claude อาจ guess ทันทีว่า "น่าจะเป็น database connection issue" แล้วเสนอ fix โดยไม่มีหลักฐาน — ซึ่งเสียเวลาถ้าเดาผิด Systematic Debugging Skill บังคับให้ gather evidence ก่อน conclude

Advanced — Custom Skill สำหรับ use case เฉพาะ

สร้าง Custom Skill ใน .claude/skills/ สำหรับองค์กร

สำหรับทีมหรือองค์กรที่ต้องการ workflow เฉพาะ — เช่น "ทุกครั้งที่สร้าง API endpoint ต้องผ่าน security checklist ของบริษัท" — คุณสร้าง Custom Skill ได้เอง

สร้างไฟล์ .claude/skills/api-security-review.md:

# API Security Review — [ชื่อบริษัท] Standard

## TRIGGER
ใช้เมื่อ: สร้าง API endpoint ใหม่, แก้ไข authentication logic, เพิ่ม data access

## WORKFLOW

### Step 1: Authentication & Authorization
- [ ] ทุก endpoint ต้องมี authentication (ยกเว้น whitelist ที่ approved)
- [ ] ตรวจ role-based access control ตาม [ชื่อบริษัท] permission matrix
- [ ] API key หรือ JWT ต้องผ่าน validation ตาม internal spec

### Step 2: Input Validation
- [ ] Validate และ sanitize ทุก input parameter
- [ ] ตรวจ SQL injection risk (ใช้ parameterized queries เสมอ)
- [ ] Rate limiting ต้องตั้งไว้ (ค่า default: 100 req/min per API key)

### Step 3: Data Exposure
- [ ] Response ต้องไม่ leak sensitive fields (password hash, internal IDs)
- [ ] ตรวจ PII handling ตาม PDPA requirements ของไทย
- [ ] Audit log ต้องบันทึกทุก data access

### Step 4: Error Handling
- [ ] Error message ต้องไม่เปิดเผย internal implementation details
- [ ] HTTP status codes ต้องถูกต้องตาม REST convention

## OUTPUT FORMAT
สรุปผลเป็น: ✅ PASS / ⚠️ NEEDS FIX / ❌ BLOCK
พร้อม issues list ที่ต้องแก้ก่อน deploy

จากนั้น invoke ด้วย:

คุณ: /api-security-review
      ช่วย review endpoint POST /users/{id}/payment-methods ที่เพิ่งสร้าง

Claude จะทำตาม checklist ของบริษัทคุณทุกครั้ง — ไม่ต้องเขียน prompt ซ้ำ และทีมทุกคนใช้ standard เดียวกันโดยอัตโนมัติ

💡
Custom Skills ที่ดี ควรมี: TRIGGER (เมื่อไรใช้), WORKFLOW ที่ชัดเจน, และ OUTPUT FORMAT ที่บอก Claude ว่าต้องส่งคืนอะไร — ถ้าขาดส่วนใดส่วนหนึ่ง Claude อาจ interpret workflow ต่างออกไป

⚠️ Common Mistakes

❌ "Skills คือ external system ที่ run แยกจาก Claude" — เหมือนมี AI อีกตัวทำงานข้างๆ
✅ Skills คือ instruction text ที่ Claude อ่านแล้ว follow ทุกอย่างยังเกิดใน Claude ตัวเดียวกัน — ไม่มี external system แยกต่างหาก
เหตุผล: ความเข้าใจผิดนี้ทำให้คิดว่า Skills มี "capability" พิเศษเกินกว่าที่ Claude ทำได้ จริงๆ แล้ว Skills แค่เปลี่ยน วิธีที่ Claude คิดและทำงาน ไม่ใช่เพิ่ม "power" ใหม่
*(source: Claude Code documentation)*
❌ "Default Claude กับ Claude+Skill ทำงานเหมือนกัน" — แค่เพิ่ม context เฉยๆ
✅ Skills เปลี่ยน workflow ทั้งกระบวนการ — Claude default จะ improvise แต่ละ request อิสระ Skills บังคับให้ทำตามขั้นตอนที่กำหนด เช่น gather information ก่อน, test before implement, verify before claim done
เหตุผล: ถ้าคิดว่าเหมือนกัน จะไม่เห็นประโยชน์ของ Skills และพลาดการใช้ workflow ที่ได้ผลดีกว่าสำหรับงาน structured เช่น debugging หรือ code review
*(source: Claude Code documentation)*
❌ "User ไม่สามารถ override Skill ได้" — ถ้า Skill บอกให้ทำอะไร Claude ต้องทำตาม Skill เสมอ
✅ User instructions มี priority สูงสุดเสมอ — ถ้าคุณบอก Claude ว่า "อย่าทำ X" แม้ว่า Skill จะบอกว่า "ต้องทำ X" Claude จะทำตามคุณ ไม่ใช่ตาม Skill
เหตุผล: Skills คือ "default recommendation" ไม่ใช่ "hard rule" คุณในฐานะ user มีสิทธิ์ตัดสินใจขั้นสุดท้ายเสมอ — นี่คือหลักการออกแบบที่ตั้งใจให้ user ควบคุม AI ได้
*(source: Superpowers documentation)*
❌ "invoke Skill ผิดแล้วงาน "พัง"" — ถ้าเลือก Skill ผิดผลลัพธ์จะแย่มาก
✅ invoke Skill ผิดแค่ทำให้ ได้ workflow ที่ไม่ optimal สำหรับงานนั้น ไม่ใช่ "พัง" เสมอ Claude ยังทำงานได้ตามปกติ แค่อาจมีขั้นตอนพิเศษที่ไม่จำเป็นสำหรับงานนั้น
เหตุผล: ไม่ต้องกลัว invoke Skill ผิด ถ้าไม่แน่ใจว่าควรใช้ Skill ไหน ลอง invoke แล้วดูว่า workflow ที่ได้ตรงกับสิ่งที่ต้องการไหม ถ้าไม่ใช่ก็บอก Claude ว่าอยากเปลี่ยน approach
*(source: Claude Code documentation)*

📝 สรุปบทที่ 4

Claude Superpowers / Skills System คือ instruction text ที่เปลี่ยน HOW Claude ทำงาน จาก "improvise ทุกครั้ง" เป็น "ทำตาม proven workflow" โดยมี priority order ที่ชัดเจน: User > Skills > Default — และ user สามารถ override ทุกอย่างได้เสมอ

🧠 ทบทวนตัวเอง — ลองตอบก่อนดูเฉลย

คำถาม 1

"ถ้า CLAUDE.md บอกว่า 'อย่าใช้ TDD' แต่ Skill test-driven-development บอกว่า 'ใช้ TDD เสมอ' — Claude จะทำอย่างไร? เพราะอะไร?"

ดูเฉลย
Claude จะทำตาม CLAUDE.md (ไม่ใช้ TDD) เพราะ priority order คือ User > Skills > Default — CLAUDE.md คือ user instruction ซึ่งมี priority สูงกว่า Skill เสมอ Skill ที่ invoke อยู่จะถูก override โดยคำสั่งของ user
คำถาม 2

"อธิบายความแตกต่างระหว่างการเขียน Prompt ยาวๆ กับการ invoke Skill สำหรับ task เดียวกัน — ทำไมอีกแบบถึงดีกว่า?"

ดูเฉลย
Prompt ยาวๆ: คุณเขียน workflow เอง อาจพลาด edge case, ตีความต่างกันแต่ละครั้ง, ต้องเขียนซ้ำทุก session

Skill: workflow ออกแบบโดย expert และทดสอบมาแล้ว, consistent ทุกครั้ง, ไม่ต้อง re-explain

Skill ดีกว่าสำหรับงาน structured ที่มี best practice ชัดเจน (เช่น debugging, code review) แต่ Prompt ยาวๆ อาจดีกว่าสำหรับงานที่ต้องการ context เฉพาะมากๆ
คำถาม 3

"Skill brainstorming และ systematic-debugging ต่างกันอย่างไร? คุณจะเลือกใช้ Skill ไหนในแต่ละสถานการณ์?"

ดูเฉลย
brainstorming: ใช้ก่อนตัดสินใจ approach หรือสร้าง feature ใหม่ — Claude จะสำรวจ intent, requirements, และ design options ก่อนลงมือ

systematic-debugging: ใช้เมื่อมี bug อยู่แล้วและต้องการหาสาเหตุ — Claude จะ gather evidence, form hypotheses, แล้วค่อย fix

เลือก brainstorming ตอน "ยังไม่มี code" เลือก systematic-debugging ตอน "code มีแล้วแต่ไม่ทำงานถูก"

⚡ ChatGPT vs Claude+Superpowers — เห็นความต่างชัดๆ

ถ้าคุณเคยใช้ ChatGPT มาก่อน นี่คือสิ่งที่ต่างกันจริงๆ เมื่อ Claude มี Skills

🐛
สถานการณ์ที่ 1: Debug โค้ดที่ผิดพลาด
มีข้อผิดพลาดใน assignment แต่หาสาเหตุไม่เจอ
❌ ChatGPT (หรือ Claude แบบ default)
คุณ: โค้ดนี้ error ทำไม?
[paste code]

AI: ปัญหาน่าจะมาจากบรรทัดที่ 12 ลองเปลี่ยน x = x + 1 เป็น x += 1 ดูนะ
⚠️ เดาสาเหตุ ไม่ได้วิเคราะห์จริง — อาจแก้ไม่หาย
✅ Claude + /systematic-debugging
คุณ: /systematic-debugging โค้ดนี้ error

Claude: ก่อนแก้ ขอ diagnose ก่อน:
1️⃣ Error message เต็มๆ คืออะไร?
2️⃣ เกิดทุกครั้งหรือบางครั้ง?
3️⃣ ก่อน error เกิด ทำอะไรล่าสุด?
4️⃣ เคยทำงานได้ก่อนหน้านี้ไหม?
✅ วินิจฉัยก่อนแก้ — แก้ได้จริงและไม่กลับมาอีก
✨
สถานการณ์ที่ 2: สร้าง Feature ใหม่
ต้องการเพิ่ม feature login ด้วย Google ใน web app
❌ ChatGPT / Claude default
AI: โอเค! มาเพิ่ม Google OAuth กัน
ขั้นตอนที่ 1: ไปที่ Google Console...
[เริ่ม implement ทันที]
⚠️ ลง code ทันทีโดยไม่รู้ stack, auth library ที่ใช้, หรือ security requirement ของโปรเจค
✅ Claude + /brainstorming
Claude: ก่อน implement ขอถามก่อน:
🔧 ใช้ framework อะไร? (Next.js, Express...)
🔑 มี auth library อยู่แล้วไหม? (NextAuth, Passport...)
💾 เก็บ session ที่ไหน? (JWT, database...)
🛡️ ต้องการ PDPA compliance ไหม?

[design ก่อน แล้วค่อย implement]
✅ รู้ context ก่อน → code ที่ได้เหมาะกับโปรเจคจริง
📝
สถานการณ์ที่ 3: วางแผนโปรเจค
ต้องทำ senior project ภายใน 3 เดือน ไม่รู้จะเริ่มจากไหน
❌ ChatGPT / Claude default
AI: วางแผนโปรเจค 3 เดือน:
เดือนที่ 1: ออกแบบ
เดือนที่ 2: พัฒนา
เดือนที่ 3: ทดสอบ
⚠️ แผน generic มากจนแทบใช้งานไม่ได้จริง
✅ Claude + /writing-plans
Claude สร้าง plan แบบ:
📁 ไฟล์ที่ต้องสร้าง/แก้ไข
✅ Task ย่อยทีละ 2-5 นาที
🧪 วิธีทดสอบแต่ละส่วน
📅 Milestone ชัดเจน
⚠️ Risk ที่ต้องระวัง
🔗 Dependencies ระหว่าง tasks
✅ แผนละเอียดพอที่จะ execute ได้จริงทันที
🔍
สถานการณ์ที่ 4: Review โค้ดก่อนส่งงาน
เขียน assignment เสร็จแล้ว อยากมั่นใจก่อนส่ง
❌ ChatGPT / Claude default
AI: โค้ดดูดีนะ! มีบางจุดที่ปรับได้:
- ตั้งชื่อตัวแปรให้ชัดขึ้น
- อาจเพิ่ม comment ได้
โดยรวมโอเคครับ 👍
⚠️ คำแนะนำผิวเผิน ไม่ตรวจ edge cases หรือ security
✅ Claude + /code-review
Claude ตรวจ systematic:
🔴 Bugs: ลืมจัดการ null pointer line 24
🟡 Performance: nested loop O(n²) ควรเป็น O(n)
🔵 Style: variable naming ไม่สม่ำเสมอ
🔒 Security: SQL query ควร parameterize
✅ ไม่มีปัญหา: logic หลักถูกต้อง
✅ Review ครอบคลุม มี severity level ชัดเจน
💬
สถานการณ์ที่ 5: ถามคำถามง่ายๆ
อยากรู้ความหมายของ API key คืออะไร
✅ ChatGPT / Claude default — เหมาะกว่า!
คุณ: API key คืออะไร?

Claude: API key คือ "รหัสลับ" ที่ใช้ระบุตัวตนเมื่อเรียกใช้ API เหมือนกับ password ที่ระบุว่า "คนนี้มีสิทธิ์ใช้บริการนี้" [อธิบายต่อ...]
✅ ได้คำตอบทันที ไม่ต้อง overhead จาก workflow
❌ ไม่จำเป็นต้อง invoke Skill
คุณ: /brainstorming API key คืออะไร?

Claude: ขอสำรวจ context ก่อน — คุณกำลัง build อะไร? ต้องการ API key ประเภทไหน? ใช้ framework อะไร?...
⚠️ Overkill — Skills ทำให้ workflow ซับซ้อนเกินความจำเป็น

🗺️ เมื่อไรควรใช้ Skills — เมื่อไรไม่ควร?

นี่คือ decision guide ง่ายๆ สำหรับมือใหม่:

✅ ควรใช้ Skills เมื่อ...
  • 🔧 งานมี best practice ที่ชัดเจน เช่น debugging, code review
  • 🔄 ทำซ้ำบ่อย — ไม่อยาก re-explain ทุกครั้ง
  • 👥 ทำงานเป็นทีม — ต้องการ standard เดียวกัน
  • 📋 มีหลาย step ที่ลืมได้ง่าย เช่น security checklist
  • 🎯 งานสำคัญ ที่ต้องการ quality สูง
❌ ไม่ต้องใช้ Skills เมื่อ...
  • ❓ คำถามง่ายๆ เช่น "อธิบาย X ให้ฉันฟัง"
  • 🎨 งาน creative ที่ต้องการ improvise
  • ⚡ ต้องการคำตอบเร็ว ไม่มี workflow ซับซ้อน
  • 🗣️ แค่อยากคุย หรือ brainstorm คร่าวๆ
  • 📖 lookup ข้อมูล ที่ไม่ต้องการ structured workflow
🧭 Rule of thumb ง่ายๆ
ถามตัวเองว่า "ถ้า expert ในด้านนี้ทำงาน พวกเขาจะทำตาม checklist หรือ process อะไร?"
ถ้าคำตอบคือ "ใช่ มี process" → ใช้ Skill
ถ้าคำตอบคือ "ไม่ แค่ตอบตรงๆ" → Claude default ก็พอ

🚀 Workflow มืออาชีพ — ทำตามได้เลย

นี่คือ workflow จริงๆ ที่นักศึกษาและ developer ใช้ทุกวัน — copy prompt แล้วใช้ได้เลย

2️⃣
วางแผน Senior Project / งานใหญ่

3️⃣

4️⃣
เขียนโค้ดแบบ TDD (Test-Driven)

Claude จะเขียน test ก่อน แล้วค่อย implement — โค้ดที่ได้จะผ่าน test ทุกตัวแน่นอน
5️⃣

6️⃣

Claude จะรัน verification commands จริงๆ และ confirm ก่อนบอกว่าผ่าน — ไม่เชื่อแค่ที่คุณบอก
