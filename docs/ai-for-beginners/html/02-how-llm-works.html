🧠 บทที่ 2

LLM ทำงานอย่างไร?

⏱ 30 นาที 🔢 Tokenization · Embedding · Attention · Generation 🧩 บทที่ 2 จาก 5

🔄 ก่อนอ่านบทนี้ — ทบทวนจากบทที่ 1

ลองตอบคำถามนี้ก่อนดูเฉลย (บังคับ! อย่าข้ามไปดูเฉลยทันที 😄):

"LLM คือส่วนหนึ่งของอะไร? และมันอยู่ในลำดับชั้นของ AI อย่างไร?"

ดูเฉลย
LLM (Large Language Model) เป็น subset ของ Deep Learning ซึ่งเป็น subset ของ Machine Learning ซึ่งเป็น subset ของ AI — ลำดับชั้นคือ AI ⊃ ML ⊃ Deep Learning ⊃ LLM

กล่าวคือ LLM คือรูปแบบหนึ่งของ Deep Learning ที่ถูก train บนข้อมูลภาษาจำนวนมหาศาล และ Deep Learning คือ ML ที่ใช้ neural network หลายชั้น ส่วน ML คือสาขาหนึ่งของ AI ที่ให้คอมพิวเตอร์เรียนรู้จากข้อมูลแทนการเขียน rules ทีละข้อ

🎯 อ่านจบบทนี้แล้ว คุณจะสามารถ...

  • อธิบายได้ว่า Tokenization คืออะไร และทำไมภาษาไทยถึงใช้ tokens มากกว่าภาษาอังกฤษในบางกรณี
  • เข้าใจว่า Embedding เปลี่ยน token เป็นตัวเลขได้อย่างไร และทำไมคำที่ "ใกล้เคียงกัน" ถึงอยู่ใกล้กันในทางคณิตศาสตร์
  • อธิบายหลักการ Attention และเหตุผลที่ทำให้ LLM เข้าใจ context ในประโยคได้
  • เข้าใจว่า Generation แบบ autoregressive ทำงานอย่างไร และทำไมจึงเห็นคำตอบทยอยออกมาทีละคำ
  • อธิบาย Context Window และผลที่มีต่อการสนทนายาวๆ
  • บอกได้ว่าทำไม LLM ถึง ไม่ได้ "ค้นหา" คำตอบ แต่ "สร้าง" มันขึ้นมา

🤔 ทำไมต้องรู้เรื่องนี้?

💡 Why This Matters

คุณเคยสงสัยไหมว่าทำไม AI ถึงบางครั้ง ตอบผิดอย่างมั่นใจมาก? หรือทำไมมันถึงลืมสิ่งที่คุยกันไปตั้งแต่ต้น conversation? หรือทำไมภาษาไทยถึงดูเหมือนยากกว่าภาษาอังกฤษสำหรับ LLM?

ถ้าคุณรู้แค่ว่า "AI ฉลาด" แต่ไม่รู้ว่ามันทำงานยังไง คุณจะใช้มันแบบ "ลุ้น" ว่าจะได้ผลหรือเปล่า แต่ถ้าคุณเข้าใจ mechanism ข้างใน คุณจะ ออกแบบการใช้งาน ให้ได้ผลดีกว่า รู้ว่าเมื่อไรควรเชื่อ เมื่อไรควร verify และทำไมบาง prompt ถึงได้ผลดีกว่าอีก prompt

💡 Analogy — มองผ่านเลนส์ที่คุ้นเคย

Analogy หลัก

LLM เหมือน autocomplete บน keyboard ของมือถือที่ฉลาดมากผิดปกติ

ลองนึกถึงตอนที่คุณพิมพ์ข้อความใน LINE แล้ว keyboard แนะนำคำที่น่าจะมาต่อ — LLM ทำงานแบบเดียวกัน แต่ฉลาดขึ้นหลายพัน (หรือล้าน) เท่า

Keyboard ของมือถือเรียนรู้จากการพิมพ์ของคุณ แต่ LLM เรียนรู้จากข้อความ เกือบทั้งหมดที่มนุษย์เขียนบนอินเทอร์เน็ต — หนังสือ บทความ code เว็บไซต์ Wikipedia รวมแล้วหลายร้อยพัน ล้านหน้า

เมื่อคุณถามอะไร มันไม่ได้ "ค้นหา" คำตอบในฐานข้อมูล แต่ ทำนายว่าคำตอบที่ดีสำหรับคำถามนี้ควรเป็นอะไร ทีละ token

⚠️ ถ้าเชื่อ analogy นี้ 100% จะเข้าใจผิดว่า:
  • LLM ไม่รู้อะไรเลย — จริงๆ แล้วมันมีความรู้มหาศาลที่ "encode" ไว้ใน weights (พารามิเตอร์หลายพันล้านตัว) เพียงแต่ความรู้นั้น implicit ไม่ใช่ตารางข้อมูล
  • LLM แค่ copy ข้อมูลที่เห็นมาแล้ว — จริงๆ แล้วมัน generalize ได้ สามารถตอบคำถามที่ไม่เคยเห็นใน training data โดยเชื่อมโยง patterns ที่เรียนรู้มา
  • LLM ทำงาน "ตัวต่อตัว" กับคำ — จริงๆ แล้วมันทำงานกับ "tokens" ซึ่งอาจเป็นส่วนของคำ, คำ, หรือหลายคำรวมกัน ไม่ใช่ตัวอักษรเดี่ยว

📖 เนื้อหาหลัก — ข้างในหัว LLM มีอะไร?

การที่ LLM สร้างคำตอบ 1 ประโยคให้คุณ ผ่านกระบวนการ 5 ขั้นตอน ลองกดดูทีละขั้น:

🔢 1. Tokenization — ภาษาผ่านสายตา AI

ก่อนที่ LLM จะทำอะไรได้ มันต้องแปลง "ข้อความ" ให้เป็นหน่วยที่จัดการได้ก่อน หน่วยนั้นเรียกว่า token

Token ไม่ใช่คำเสมอไป — มันอาจเป็น:

  • ทั้งคำ เช่น "cat" = 1 token
  • ส่วนของคำ เช่น "playing" = "play" + "ing" = 2 tokens
  • เครื่องหมายวรรคตอน เช่น "!" = 1 token
  • ช่องว่างรวมกับคำ เช่น " Hello" (มี space นำหน้า) = 1 token
🇹🇭
ทำไมภาษาไทยถึงใช้ tokens มากกว่า?
LLM ส่วนใหญ่ train บน data ภาษาอังกฤษมากกว่า ทำให้ tokenizer "รู้จัก" คำศัพท์ภาษาอังกฤษเป็นหน่วย แต่ภาษาไทยมักถูกแบ่งเป็น subunit เล็กกว่า ส่งผลให้ข้อความไทยความยาวเท่ากันอาจใช้ tokens มากกว่า 2-3 เท่า — ซึ่งกินพื้นที่ context window มากกว่า และมีต้นทุนสูงกว่า

ลองเล่น Token Visualizer ด้านล่าง — พิมพ์ข้อความแล้วดูว่าถูกแบ่งเป็น tokens ยังไง:

ℹ️
หมายเหตุ: Token visualizer ด้านบนเป็น approximation เพื่อแสดงแนวคิด tokenizer จริงๆ ของ Claude ใช้ BPE (Byte-Pair Encoding) ซึ่งซับซ้อนกว่านี้ โดยประมาณ 1 token ≈ 4 ตัวอักษรภาษาอังกฤษ

🌐 2. Embedding — แปลงความหมายเป็นตัวเลข

หลังจากได้ tokens แล้ว แต่ละ token ถูกแปลงเป็น vector — ชุดตัวเลขหลายพันมิติที่จับ "ความหมาย" ของ token นั้นไว้

ลองนึกถึง coordinate บนแผนที่ — เมืองที่อยู่ใกล้กันบนแผนที่มี coordinates ที่ใกล้กัน ในโลกของ Embedding: คำที่มีความหมายใกล้กันจะมี vectors ที่อยู่ใกล้กัน

ตัวอย่าง Word Vector Arithmetic (ที่มีชื่อเสียงมาก)

"กษัตริย์" − "ชาย" + "หญิง" ≈ "ราชินี"

นี่คือ word2vec example ที่โด่งดัง — แสดงให้เห็นว่า vectors จับ "ความสัมพันธ์เชิงความหมาย" ไม่ใช่แค่ความคล้ายของตัวอักษร ซึ่งหมายความว่า Embedding เข้าใจ concept เช่น เพศ, สถานะทางสังคม, ความเกี่ยวข้อง

🔍 3. Attention — โฟกัสส่วนที่สำคัญ

ปัญหาของการเข้าใจภาษาคือ ความหมายขึ้นอยู่กับ context ลองดูตัวอย่าง:

บริบท: แมวกำลังกินอาหาร

"มันกินมัน เพราะมันหิว"

  • "มัน" ตัวแรก = แมว (subject — ผู้กระทำ)
  • "มัน" ตัวที่สอง = อาหาร (object — สิ่งที่ถูกกิน)
  • "มัน" ตัวที่สาม = แมว (อีกครั้ง — ผู้ที่หิว)

ทั้งสามคำเป็น "มัน" เหมือนกัน แต่มีความหมายต่างกัน — Attention mechanism ช่วยให้ model "โฟกัส" ไปที่ tokens ที่เกี่ยวข้องกัน เมื่อประมวลผล "มัน" ตัวที่สาม model ให้ attention มากกับ "มัน" ตัวแรก + "กิน" มากกว่า tokens อื่น จึงรู้ว่าหมายถึง "แมว"

Transformer architecture (ที่ LLM ทุกตัวใช้) ทำ Attention นี้ หลายร้อยครั้งพร้อมกัน ในแต่ละ layer แต่ละครั้งมอง "ความสัมพันธ์" คนละมิติ เช่น ความสัมพันธ์เชิงไวยากรณ์, เชิงความหมาย, เชิง coreference ฯลฯ

✨ 4. Generation — สร้างคำตอบทีละ Token

นี่คือขั้นตอนที่น่าทึ่งที่สุด — LLM ไม่ได้ค้นหาคำตอบ แต่สร้างมันขึ้นมา ทีละ token

กระบวนการคร่าวๆ:

  1. รับ context ทั้งหมด (prompt + tokens ที่สร้างมาแล้ว)
  2. คำนวณ probability distribution ของ "token ที่น่าจะมาต่อ"
  3. เลือก token หนึ่งตัว (แบบ probabilistic — ไม่ได้เลือก highest probability เสมอไป)
  4. append token นั้นเข้าไปใน context
  5. วนซ้ำตั้งแต่ข้อ 1 จนกว่าจะจบ
🌡️
Temperature คืออะไร?
Temperature ส่งผลต่อขั้นตอนที่ 3 — Temperature สูง = เลือก tokens ที่มี probability ต่ำกว่าบ้าง → ได้คำตอบที่ creative/แปลกใหม่ กว่า | Temperature ต่ำ = เลือก token ที่มี probability สูงที่สุดเสมอ → ได้คำตอบที่ predictable/คงที่กว่า นี่คือเหตุผลที่ creative writing ควรใช้ temperature สูงกว่า ส่วนงาน factual/code ควรใช้ต่ำกว่า

📦 Context Window — ความจำระยะสั้นของ LLM

Context Window คืออะไร

ทุกอย่างใน conversation ปัจจุบัน — ทั้ง prompt ของคุณ, คำตอบก่อนหน้า, system instructions — ถูก "อ่าน" ทุกครั้งที่ LLM จะตอบ

มีขีดจำกัดของ size ที่เรียกว่า Context Window (วัดเป็น tokens) เช่น 200,000 tokens สำหรับ Claude บางรุ่น

ผลต่อการใช้งาน

เมื่อจบ session: ทุกอย่างถูกลืม ไม่มี persistent memory ข้าม sessions

เมื่อ conversation ยาวเกิน context: ส่วนแรกถูกตัดออก ทำให้ LLM "ลืม" บทสนทนาช่วงต้น

🔑
จำไว้เสมอ: Context Window ≠ Memory ถาวร — ถ้าต้องการให้ AI "จำ" ข้อมูลสำคัญข้าม sessions ต้องมี external memory system ที่ผู้พัฒนา implement เพิ่มเติม ไม่ใช่ feature ที่มีอยู่ใน LLM โดย default

📚 ตัวอย่าง 3 ระดับ

Beginner — Google Translate กับ LLM

Google Translate ใช้ LLM concepts อย่างไร?

เมื่อคุณ translate "I love learning" เป็นภาษาไทย Google Translate ต้องผ่านกระบวนการที่คล้ายกับที่เพิ่งอ่านมา:

🔢
Tokenization
"I", " love", " learning" ถูกแยกเป็น 3 tokens แยกกัน
🌐
Embedding
แต่ละ token ถูกแปลงเป็น vector ที่จับความหมายของมัน
🔍
Attention
Model รู้ว่า "learning" สัมพันธ์กับ "love" — ฉันรัก "การเรียนรู้" ไม่ใช่รัก "กำลังเรียน"
✨
Generation
สร้าง "ฉัน", "รัก", "การเรียนรู้" ทีละ token ในภาษาปลายทาง

สิ่งที่น่าสนใจคือ — Google Translate รุ่นเก่า (pre-2016) ไม่ได้ใช้ Transformer และผลลัพธ์แย่กว่ามาก โดยเฉพาะกับประโยคยาวที่ต้องเข้าใจ context เพราะไม่มี Attention mechanism

Intermediate — ทำไมภาษาไทยถึงมีปัญหาเรื่องการสะกด

ทำไม Claude ตอบภาษาไทยได้ แต่บางครั้งสะกดผิดหรือใช้คำแปลกๆ?

นี่ไม่ใช่เรื่องบังเอิญ — มีเหตุผลที่อธิบายได้ผ่าน 2 concepts:

เหตุผลที่ 1: Training Data Distribution

ข้อมูลบนอินเทอร์เน็ตเป็นภาษาอังกฤษมากกว่าภาษาไทยอย่างมีนัยสำคัญ (ประมาณ 50%+ ของ CommonCrawl เป็นภาษาอังกฤษ ส่วนภาษาไทยน้อยกว่า 1%) ทำให้ LLM เห็น "ตัวอย่าง" ภาษาอังกฤษที่ถูกต้องมากกว่ามาก → Probability distribution สำหรับ next token ในภาษาไทยมี "ความแม่นยำ" น้อยกว่า

เหตุผลที่ 2: Tokenization Mismatch

Tokenizer ที่ optimize สำหรับภาษาอังกฤษจะแบ่งภาษาไทยอย่าง "หยาบ" กว่า เช่น:

  • คำว่า "สวัสดี" อาจถูกแบ่งเป็น bytes หรือ character ย่อยๆ แทนที่จะเป็น 1 token
  • ทำให้ model เห็น pattern ระดับ character มากกว่า pattern ระดับ word
  • เมื่อ generate ออกมา อาจเกิด character combination ที่ผิดหลักภาษา

ผลลัพธ์: LLM "รู้" ภาษาไทย แต่ความ "ลื่นไหล" และความถูกต้องน้อยกว่าภาษาอังกฤษ

Advanced — Hallucination เกิดจาก Generation Process อย่างไร?

ทำไม LLM ถึงบางครั้ง "โกหกอย่างมั่นใจ" (Hallucinate)?

Hallucination ไม่ใช่ bug — มันคือ emergent behavior ที่เข้าใจได้จาก generation mechanism:

Root Cause: Confident Next-Token Prediction ที่ผิด

เมื่อ LLM generate คำตอบ มันคำนวณ probability ของ next token จาก patterns ใน training data ปัญหาเกิดเมื่อ:

  1. Pattern ที่ดูเหมือนถูกต้อง แต่ facts ผิด — เช่น "นักวิทยาศาสตร์ X ค้นพบ Y ในปี Z" อาจ generate ได้ดูถูกต้องตามรูปแบบ แต่ X, Y, Z อาจผสมกันผิด
  2. Confident generation บน low-frequency information — ข้อมูลที่ปรากฏน้อยใน training data มี probability distribution ที่กระจาย model จึง generate บน "best guess" ที่อาจผิด
  3. Autoregressive error accumulation — เมื่อ token แรกผิด token ถัดไปก็ generate จาก context ที่มี token ผิดนั้น ทำให้ errors สะสม
⚠️
นัยสำคัญ: LLM ไม่ "รู้ว่าตัวเองไม่รู้" เพราะ generation process คำนวณ probability เสมอ ไม่มี mechanism ที่บอกว่า "ฉันไม่มีข้อมูลนี้" — นั่นคือเหตุผลที่ต้อง verify ข้อมูล factual สำคัญเสมอ

🚫 Common Mistakes — ความเข้าใจผิดที่พบบ่อย

❌ "LLM ค้นหาข้อมูลจาก internet ตลอดเวลาที่ตอบ"
✅ LLM ทำงานจาก weights (พารามิเตอร์) ที่ train ไว้แล้ว ไม่ได้ browse internet ขณะตอบ — ความรู้ถูก "encode" ลงใน weights ณ เวลา training ยกเว้นมี tool ที่ให้ทำ (เช่น web search tool ที่ต้องมี developer เปิดใช้งาน)
เหตุผลที่เข้าใจผิด: เพราะ AI ดูเหมือน "รู้ทุกอย่าง" แต่ความรู้นั้นมาจาก training data ที่ตัดรอบในวันที่หนึ่ง (knowledge cutoff)
*(source: Anthropic model documentation)*
❌ "LLM จำทุกอย่างไว้ตลอดชีวิต — มันจะ 'รู้' ว่าเราเป็นใครทุกครั้งที่คุย"
✅ ข้อมูลอยู่ใน context window เท่านั้น เมื่อจบ session = ลืมทุกอย่าง ในรุ่นปัจจุบัน (Claude.ai) มี Projects ที่เพิ่ม persistent context ได้ แต่นั่นคือ feature ที่ต้องตั้งค่าแยกต่างหาก ไม่ใช่ default behavior ของ model
เหตุผลที่สำคัญ: ถ้าแชร์ข้อมูลส่วนตัวโดยคิดว่า AI จะ "จำ" ข้ามรอบ อาจทำให้ประเมินความเป็นส่วนตัวผิดพลาด
*(source: Claude documentation — How Claude works)*
❌ "LLM คิดเหมือนสมองคน — มันเข้าใจ ใคร่ครวญ และมีเหตุผลเหมือนมนุษย์"
✅ LLM ทำ statistical pattern matching ที่ซับซ้อนมาก แต่ต่างจาก human reasoning อย่างมาก — ไม่มี "ความเข้าใจ" เชิง conceptual แบบที่มนุษย์มี, ไม่มี common sense grounding จากประสบการณ์โลกจริง, และไม่ได้ "คิด" ก่อนตอบแบบที่คนทำ (มัน generate ทีละ token ไปข้างหน้า)
เหตุผลที่สำคัญ: การคาดหวังว่า AI จะ "คิดแบบคน" ทำให้ประเมิน capability ผิด ทั้ง overestimate (เชื่อทุกอย่าง) และ underestimate (ไม่รู้ว่าใช้ได้ดีแค่ไหน)
*(source: AI research community — understanding LLM capabilities and limitations)*

📝 สรุปบท — สิ่งสำคัญที่ต้องจำ

🔢 Tokenization
ข้อความถูกแบ่งเป็น tokens (ไม่ใช่คำ) — ภาษาไทยใช้ tokens มากกว่าภาษาอังกฤษ
🌐 Embedding
Token → Vector ที่จับความหมาย คำที่ใกล้กันมี vectors ใกล้กัน
🔍 Attention
Model โฟกัส tokens ที่เกี่ยวข้องกัน ทำให้เข้าใจ context ในประโยคได้
✨ Generation
สร้างคำตอบทีละ token แบบ probabilistic — ไม่ได้ "ค้นหา" แต่ "สร้าง"

กฎทอง: LLM ไม่ได้รู้คำตอบ แต่ทำนายว่า token อะไรน่าจะมาต่อ — นั่นคือเหตุผลที่มันเก่งมากในบางเรื่อง และผิดอย่างมั่นใจในบางเรื่อง Context Window คือขีดจำกัดของ "ความจำ" ในแต่ละ session

🧠 Retrieval Questions — ลองตอบก่อนดูเฉลย
คำถามที่ 1

"อธิบาย tokenization ให้คนที่ไม่รู้อะไรเกี่ยวกับ AI ฟัง โดยใช้ analogy ที่ไม่มีในบทเรียนนี้"

ดูแนวคำตอบ
ตัวอย่าง analogy: "เหมือน LEGO — ข้อความทั้งประโยคไม่ได้ถูกมองเป็นชิ้นเดียว แต่ถูกแยกออกเป็นชิ้น LEGO ย่อยๆ ที่เรียกว่า tokens บางชิ้นเป็นคำสั้น บางชิ้นเป็นส่วนของคำ AI ทำงานกับ LEGO แต่ละชิ้น ไม่ใช่ภาพรวมทั้งหมด" — สำคัญที่ต้องอธิบายคือ token ≠ ตัวอักษร และ token ≠ คำทั้งคำ
คำถามที่ 2

"ทำไม LLM ถึงตอบในภาษาไทยได้แย่กว่าภาษาอังกฤษในบางกรณี — อธิบายผ่าน concept tokenization และ training data"

ดูแนวคำตอบ
สองเหตุผลหลัก: (1) Training data ภาษาอังกฤษมีมากกว่าภาษาไทยมาก ทำให้ probability distribution สำหรับ next token ในภาษาไทยแม่นยำน้อยกว่า (2) Tokenizer ที่ design มาสำหรับภาษาอังกฤษแบ่งภาษาไทยเป็น unit เล็กกว่า ทำให้ model เห็น pattern ภาษาไทยในระดับ sub-word มากกว่า word — ผลลัพธ์คือการ generate ภาษาไทยมีโอกาสสะกดผิดหรือใช้คำแปลกๆ มากกว่า
คำถามที่ 3

"Context window คืออะไร และมันส่งผลต่อการสนทนายาวๆ อย่างไร?"

ดูแนวคำตอบ
Context window คือขนาดสูงสุดของ tokens ที่ LLM สามารถ "อ่าน" ได้ในการตอบครั้งหนึ่ง ผลต่อสนทนายาวๆ: (1) เมื่อ conversation ยาวเกิน context window ส่วนแรกถูกตัดออก ทำให้ LLM "ลืม" บทสนทนาช่วงต้น (2) เมื่อจบ session ทุกอย่างถูก reset — LLM ไม่มี persistent memory ข้าม sessions (3) ยิ่ง conversation ยาว ยิ่งกิน tokens มาก และมีต้นทุนสูงกว่า
